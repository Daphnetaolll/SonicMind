## Technical details

The initial SonicMind prototype uses a classic RAG flow:

```text
user question
-> question embedding
-> vector retrieval from the knowledge base
-> reranking
-> selected evidence chunks
-> LLM-grounded answer generation
-> conversation memory update
```

The knowledge base was designed around semantically meaningful chunks rather than raw documents. Source texts were cleaned, split by Chinese punctuation, grouped into roughly 350-character chunks, and overlapped with sentence-level fallback to reduce context breaks. This produced 129 chunks in `data/processed/chunks.jsonl`.

For retrieval, `moka-ai/m3e-base` was selected because it is optimized for Chinese semantic matching. The vectors were L2-normalized, allowing FAISS inner product search to approximate cosine similarity. The resulting index is stored as `data/index/faiss.index`.

The RAG pipeline exposes a unified `answer_question(query, chat_history=None, ...)` entry point. It returns the final answer, original retrieved candidates, reranked documents, final documents used for generation, and query rewrite metadata.

The Streamlit interface made the system demonstrable by combining local knowledge base usage, optional document upload, rebuild controls, answer rendering, reference snippets, and session-based conversation history.

## Issues / Bugs

- Early retrieval-only answers were relevant but not always clear, complete, or logically organized.
- Origin and genre-list questions sometimes returned partially relevant chunks without enough synthesis.
- The small three-document dataset limited semantic coverage and increased chunk redundancy.
- Follow-up questions originally drifted during retrieval because memory was only used during generation, not during retrieval.
- The web app initially failed when the LLM environment variable was missing.

## Solutions / Decisions

- Decided to connect retrieval with LLM generation so the system could synthesize answers rather than only return matching chunks.
- Added prompt constraints requiring the answer to rely on knowledge base evidence and admit uncertainty when the current material is insufficient.
- Added history-aware query rewriting before retrieval so follow-up questions could resolve references such as "it", "this style", or "that genre".
- Separated reranking into `src/reranker.py` so retrieval quality could be improved independently of generation.
- Used larger initial candidate pools and then trimmed to the final top-k evidence after reranking.
- Made the web app usable with the bundled local knowledge base by default, while keeping document upload as an expansion path.
- Documented that LLM environment variables must be set in the same terminal session used to start Streamlit.

## Next steps

- Replace character-based chunking with token-aware chunking for more consistent context windows.
- Expand the knowledge base with more electronic music genres, artists, labels, scenes, and historical sources.
- Introduce a stronger cross-encoder reranker for higher-quality evidence ordering.
- Consider IVF or HNSW indexing when the corpus becomes larger.
- Improve answer evaluation with more structured test cases for definitions, comparisons, history, recommendations, and multi-turn references.
- Continue evolving the Streamlit prototype toward a production web architecture.

## Notes / Reflection

This development pass established the foundation of SonicMind: a source-grounded music assistant that can retrieve Chinese electronic music knowledge, generate answers from evidence, remember short conversation context, and expose the workflow through a usable web interface. The most important architectural lesson was that retrieval, generation, memory, and UI are not separate demos; they need to work together as one loop for the assistant to feel coherent.
