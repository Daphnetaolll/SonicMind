## Technical details

The billing layer now separates first-time subscription purchase, subscription management, and direct plan upgrades:

- First-time Creator/Pro purchases use `POST /api/billing/checkout-session`.
- Payment methods, invoices, cancellation, and downgrades use `POST /api/billing/portal-session`.
- Creator to Pro upgrades use `POST /api/billing/subscription-plan`.

The direct upgrade endpoint reads the current Stripe-backed provider subscription from PostgreSQL, retrieves the Stripe subscription, identifies the existing subscription item, and calls Stripe's subscription update API with that item id and the Pro price id. This prevents double billing from accidentally adding a second subscription item.

The account status response is now centralized so `/api/me` and billing plan-change responses both return the same shape. The frontend can update pricing and chat badges from one account-status payload.

The deployed backend remains in `production_light` mode with lexical retrieval, one Uvicorn worker, and no heavy FAISS/torch/sentence-transformers runtime dependency. Semantic FAISS mode remains documented for local or future high-memory deployment.

## Issues / Bugs

- A user who canceled Creator could still have an active Stripe subscription until the period end, so the app treated them as paid and sent all paid-plan buttons to Billing Portal.
- The frontend had no direct path for Creator users to upgrade to Pro from the SonicMind pricing page.
- Webhook responses could report `processed=true` even if no local paid subscription was actually synced.
- The topbar and pricing plan display could remain stale if `users.plan` had not caught up with quota's provider-backed subscription check.
- Project documentation still contained older Streamlit-first language and placeholder deployment URLs.

## Solutions / Decisions

- Keep Billing Portal for cancellation, invoices, payment methods, and downgrades because Stripe owns those flows best.
- Add direct in-app upgrade only for Creator to Pro, where the product intent is clear and it is safer to replace the existing subscription item than create a new Checkout session.
- Keep price ids and Stripe secrets backend-only.
- Use metadata fallback for plan detection, but only accept supported plan codes (`creator` and `pro`).
- Document production lightweight retrieval as the deployed default because it keeps the Render 2 GB backend stable.
- Treat the legacy Streamlit app as a comparison path, not the primary deployed product.

## Notes / Reflection

Today's work clarified an important billing boundary: Checkout is for starting subscriptions, Portal is for broad subscription management, and direct plan-change API calls are appropriate only when the app can safely identify and modify one existing subscription item.
