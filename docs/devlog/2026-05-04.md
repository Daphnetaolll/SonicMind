# Devlog — 2026-05-04

## Goal

Prepare SonicMind for a public beta / portfolio review by moving the app from a Streamlit-only prototype toward a React + FastAPI architecture while preserving the existing RAG, auth, quota, and music recommendation behavior.

## Work completed

- Separated reusable backend logic from the Streamlit UI into service modules for accounts, chat, errors, tokens, and knowledge-base administration.
- Added a FastAPI backend with health, login, register, and authenticated chat endpoints.
- Added a Vite + React frontend with login, register, chat, source inspection, Spotify cards, and retrieval settings.
- Updated the README with React/FastAPI setup instructions, backend secret guidance, migration status, and known limitations.
- Improved music intent routing for mood-based recommendations, common techno misspellings, DJ-set track requests, and bilingual genre questions.
- Added fallback behavior so Spotify recommendation cards can use curated genre examples when live/current-source discovery cannot resolve exact tracks.
- Added API and music-intent tests for endpoint serialization, sanitized storage errors, Spotify hiding behavior, and Spotify track embed URL formatting.
- Documented deployment and QA notes in project docs.

## Technical details

The Streamlit app now calls shared backend service functions instead of owning account validation, knowledge-base rebuilds, question logging, and quota charging directly. FastAPI exposes the same core behavior through JSON-safe schemas, with bearer-token auth handled by a lightweight backend-only HMAC token during this migration stage.

The React frontend uses React Router for pages, axios for API calls, TanStack Query for request state, zustand for persisted auth/conversation state, and React Bootstrap for the interface. The chat page keeps the latest answer, remaining quota, source diagnostics, Spotify cards, and retrieval controls visible in one workflow.

Music routing now distinguishes explanation, comparison, label/artist/profile requests, and playable track recommendations more carefully. Mood hints are mapped to broad electronic genres only as routing hints, not as definitive user preferences.

## Issues / Bugs

- The Streamlit prototype had UI-specific validation and knowledge-base logic that would have drifted from the new API without a shared service layer.
- Browser clients must not receive backend secrets such as LLM keys, Spotify secrets, database URLs, or token signing secrets.
- Spotify searches may return playable but weak matches when the user asks for current or mood-based recommendations.
- Full local verification depends on configured database, LLM, Spotify, and search credentials.

## Solutions / Decisions

- Kept secrets and durable operations in backend services and exposed only safe response models to the React app.
- Preserved the Streamlit app as a legacy comparison path while adding FastAPI and React as the migration target.
- Used explicit API schemas to serialize rich RAG results into frontend-ready answer, source, music, Spotify, history, and quota fields.
- Added source-grounded recommendation planning before Spotify display so playable cards stay connected to evidence.
- Documented the current migration phases and limitations instead of implying production readiness.

## Notes / Reflection

The main architectural shift was moving SonicMind from a single Streamlit surface into a reusable backend plus a browser frontend. That makes the project easier to deploy, test, and present, while keeping the original RAG pipeline available. The biggest tradeoff is that the system now has more moving parts, so documentation and clear service boundaries matter more than they did in the prototype.
