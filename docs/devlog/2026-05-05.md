# Devlog — 2026-05-05

## Goal

Run a deeper QA and polish pass on the SonicMind React + FastAPI beta so the project is stronger for portfolio review, especially around recommendation quality, public-facing pages, multilingual behavior, and failure handling.

## Work completed

- Ran an 80-question API QA pass across music knowledge, scenes, recommendations, DJ/playlist prompts, Chinese and mixed-language prompts, and edge cases.
- Added a real public landing page instead of sending visitors straight into the app flow.
- Polished the pricing page and preserved clear Coming Soon behavior for upgrade and extra-pack actions.
- Improved playlist and DJ-set routing so build/set/playlist prompts are handled more intentionally.
- Strengthened Chinese and mixed-language answer behavior with explicit synthesis instructions and structured fallback text.
- Added SonicMind-owned support responses for pricing, login/password, and private-information safety prompts.
- Added Spotify access-token caching, search-result caching, and provider backoff handling after live QA triggered Spotify HTTP 429 rate limiting.
- Changed empty chat input handling so the API returns the intended clean `400` validation response instead of a Pydantic `422`.
- Verified backend health, pricing endpoint behavior, frontend build, backend tests, and browser routes for `/`, `/pricing`, and `/chat`.
- Documented the QA results and remaining risks in `docs/AUTONOMOUS_QA_POLISH_REPORT.md`.

## Technical details

The QA pass exercised the live `POST /api/chat` flow as a Pro test account and scored answer quality, Spotify relevance, issue patterns, fixes, and retest results. Most text-answer paths held up well, while the biggest repeated failure mode came from live Spotify lookups exhausting provider limits during the long run.

Spotify handling now reduces repeated provider calls by caching access tokens and search results, and it backs off cleanly when the provider returns rate-limit responses. Recommendation text can still succeed when Spotify cards are temporarily unavailable.

Support-style prompts now use local SonicMind responses for product-specific questions instead of relying on broader retrieval sources that can confuse the app with Spotify's own support content. Chinese and mixed-language prompts also receive stronger language guidance during answer synthesis.

## Issues / Bugs

- Spotify live catalog calls hit HTTP 429 during the long QA run.
- Chinese and mixed-language prompts sometimes answered in English.
- Product support prompts could be answered from unrelated web or Spotify support evidence.
- Empty chat input returned a framework-level `422` instead of the app's cleaner validation message.
- Playlist and DJ-set prompts needed clearer routing separate from direct track recommendation prompts.
- Spotify cards could not be fully retested while the provider rate limit was still active.

## Solutions / Decisions

- Added Spotify token/search caching and backoff behavior to reduce repeated provider pressure and degrade gracefully after rate limiting.
- Added explicit Chinese and mixed-language instructions near answer synthesis.
- Added local support/safety responses for SonicMind pricing, login/password, and private-information prompts.
- Relaxed schema-level empty-input validation so the route can return a clean `400` response with `Question is required.`
- Treated build/set/playlist prompts as playlist-style requests while preserving explicit song/track recommendation routing.
- Documented Spotify rate limiting as an external-provider risk instead of treating it as a text-answer failure.
