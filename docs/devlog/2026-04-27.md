# Devlog — 2026-04-27

## Goal

Prepare the project for a public-facing music RAG web app with controlled usage, account-based access, hybrid retrieval, and Spotify display that stays aligned with source-grounded answers.

## Work completed

- Standardized local setup files, environment variable documentation, and Streamlit startup instructions.
- Added PostgreSQL-backed account, quota, usage, admin, and local subscription scaffolding.
- Converted the UI, prompts, fallback messages, and knowledge base content direction to English.
- Rebuilt the RAG architecture from local-only retrieval into a hybrid flow: local knowledge base, trusted music sources, then Tavily web search.
- Added trusted music source policy for MusicBrainz, Discogs, Every Noise, Ishkur, Rate Your Music, AllMusic, Resident Advisor, Beatport, Traxsource, Billboard, Official Charts, DJ Mag, and Mixmag.
- Reworked Spotify integration so Spotify is a display layer, not the recommendation engine.
- Added dynamic music recommendation planning so text answers and Spotify cards share the same source-grounded candidate tracks.
- Added stricter filtering for bad music candidates such as compilations, forum threads, chart page titles, and generic list pages.
- Added curated fallback examples for a few known styles while moving the main recommendation path toward dynamic trusted-source discovery.

## Technical details

The app now separates three responsibilities:

- RAG evidence retrieval decides what information is available.
- The music recommendation planner extracts and ranks candidate tracks from trusted evidence and web search.
- Spotify resolves already-selected `title + artist` candidates into playable cards.

For recent or trending track questions, Spotify no longer falls back to old curated classics when dynamic candidates fail validation. This prevents the answer from discussing current music while the UI displays unrelated historical tracks.

The current dynamic flow is:

```text
user question
-> query understanding
-> hybrid evidence routing
-> music recommendation plan
-> exact Spotify title+artist validation
-> answer and Spotify cards from the same candidate list
```

## Issues / Bugs

- `who is John Summit` is currently classified as general music knowledge instead of an artist/entity profile.
- `tell me about John Summit` can be misclassified as a genre explanation because unknown names are treated too eagerly as genre hints.
- Dynamic web extraction can still miss good results when source pages do not expose clear `artist - track` or `track · artist` patterns.
- Spotify API search is useful for catalog resolution, but it cannot reliably decide what is trending or representative by itself.

## Solutions / Decisions

- Decided that Spotify must remain display-only and must not decide recommendation quality.
- Introduced a shared recommendation plan so Answer and Spotify Matches use the same candidate list.
- Added stricter source parsing for Beatport and Traxsource style snippets.
- Prevented recent/trending track questions from falling back to curated examples when dynamic candidates fail.
- Identified the next architectural fix: add profile subject extraction and evidence-based entity type resolution for `who is X`, `tell me about X`, and similar artist/entity questions.

## Next steps

- Add `profile subject extraction` for questions like `who is John Summit`, `who is Peggy Gou`, and `tell me about Skrillex`.
- Resolve ambiguous subjects with evidence and Spotify search before deciding whether they are artist, label, genre, album, track, or playlist.
- Route resolved artists to Spotify artist cards plus validated top or representative tracks.
- Add regression tests for artist profile, label profile, genre explanation, trending tracks, and ambiguous entity questions.
- Continue reducing dependence on manually curated recommendation JSON by improving dynamic trusted-source extraction.

## Notes / Reflection

The biggest design shift today was moving from "Spotify recommends music" to "trusted evidence recommends music, Spotify displays it." That matters because Spotify search can return playable but semantically wrong results. The more reliable architecture is to ground recommendations first, then use Spotify only for validated playback links and metadata.
