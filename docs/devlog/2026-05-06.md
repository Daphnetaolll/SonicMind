# Devlog — 2026-05-06

## Goal

Stabilize the deployed SonicMind backend after Render production testing exposed chat failures, Spotify display issues, and memory pressure from semantic retrieval dependencies. The main goal was to keep the current 2 GB Render backend reliable while preserving a higher-quality semantic retrieval path for local or future larger deployments.

## Work completed

- Investigated production chat failures after deployment and traced the critical backend crash to Render memory limits during normal chat usage.
- Added a clear two-mode runtime architecture: production lightweight lexical mode and local/high-memory semantic FAISS mode.
- Created a central runtime settings layer for retrieval backend, embedding model usage, reranker usage, RAG limits, fallback behavior, and memory-sensitive defaults.
- Updated `/api/health` to report safe, non-secret diagnostics such as active mode, retrieval backend, knowledge-base readiness, semantic readiness, and heavy dependency availability.
- Kept production mode compatible with `requirements-production.txt`, which excludes FAISS, torch, transformers, and sentence-transformers.
- Preserved the existing FAISS and `BAAI/bge-m3` semantic path behind explicit environment flags for local testing or future larger Render instances.
- Added lightweight and semantic backend launch scripts under `scripts/`.
- Expanded `scripts/memory_probe.py` so it can measure both lexical and FAISS modes.
- Added mode-specific environment example files for production lightweight, local lightweight, and local semantic runs.
- Updated Render deployment documentation with exact build command, start command, environment variables, runtime-mode guidance, and memory probe notes.
- Improved the production smoke test so it verifies the active retrieval backend reported by health.
- Fixed a recommendation parsing issue where `recommend me some popular dance music recently` was interpreted as the malformed genre `some popular dance`.
- Added curated fallback recommendations for `electronic dance music` and improved answer synthesis so generic "I could not find tracks" text is replaced with concrete source-grounded candidates when available.

## Technical details

The new runtime configuration lives in `src/settings.py`. It resolves `APP_ENV`, `SONICMIND_MODE`, `SONICMIND_RETRIEVAL_BACKEND`, `ENABLE_LOCAL_EMBEDDING_MODEL`, `ENABLE_RERANKER`, `RAG_LOAD_ON_STARTUP`, RAG size limits, context limits, and fallback behavior from one place. Explicit environment variables win over defaults.

Production lightweight mode defaults to lexical retrieval with small retrieval and prompt budgets. In this mode, FastAPI app import and health checks do not import FAISS, torch, transformers, or sentence-transformers. The active knowledge-base readiness check only requires `chunks.jsonl` and `chunk_meta.jsonl`.

Semantic mode remains available through `SONICMIND_RETRIEVAL_BACKEND=faiss` and `ENABLE_LOCAL_EMBEDDING_MODEL=true`. FAISS index loading and sentence-transformer construction are still lazy and cached so they happen only on semantic retrieval paths instead of every request.

The health endpoint now exposes runtime state without secrets. It reports whether semantic dependencies are available by using import metadata checks instead of importing heavy packages.

For recommendation quality, the query-understanding layer now normalizes `dance music` and `EDM` to `electronic dance music`, strips filler words such as `some`, `popular`, and `recently`, and maps `dnb` / `drum & bass` to `drum and bass`. Recommendation planning now falls back to curated representative tracks when trusted-source search cannot extract exact current chart candidates.

## Issues / Bugs

- Render reported that the backend instance ran out of memory over the 2 GB limit during chat.
- The deployed chat path could cold-load semantic retrieval dependencies that are too heavy for the current production backend.
- `/api/health` previously did not expose enough mode-specific diagnostics for production debugging.
- Production readiness previously treated FAISS files as required even when lexical mode was selected.
- Spotify cards could be hidden in production when backend Spotify credentials were missing or unavailable.
- A recent dance music recommendation prompt produced an off-topic answer about house music because the genre parser extracted a noisy genre hint.
- Semantic memory probing locally loaded `BAAI/bge-m3` to roughly 959 MB RSS and then the process terminated before completing retrieval.

## Solutions / Decisions

- Made `production_light` + `lexical` the safe deployed default for Render 2 GB.
- Kept semantic FAISS mode intact, but made it explicit and opt-in for local/high-memory environments.
- Split dependency profiles into `requirements-production.txt` for lightweight production and `requirements.txt` for full semantic development.
- Added `WEB_CONCURRENCY=1` and native thread-limiting environment variables to the documented Render production profile.
- Changed knowledge-base readiness so lexical mode does not fail just because FAISS or sentence-transformers is unavailable.
- Added safe mode and memory logs such as `[MODE]` and `[RETRIEVAL]` without logging secrets, tokens, database URLs, or user credentials.
- Added tests proving lightweight backend import does not load heavy semantic modules.
- Treated semantic FAISS on Render as a future 4 GB or 8 GB deployment option, not the current default.
- Made recommendation answers align with the structured track candidates that Spotify cards would display.

## Notes / Reflection

Today clarified that SonicMind needs two honest operating modes instead of one mode trying to do everything. The lightweight production mode protects the deployed demo from memory crashes, while semantic mode keeps the more ambitious retrieval architecture available for higher-memory environments. The tradeoff is acceptable for the current beta: production stays stable and cheap, and semantic retrieval remains a documented upgrade path rather than a hidden source of outages.
